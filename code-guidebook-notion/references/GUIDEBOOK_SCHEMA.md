# 가이드북 spec JSON 스키마

각 노트북 → 가이드북 1개. `_guidebook/gb_<slug>.json` 에 저장.

```jsonc
{
  "title": "📘 가이드북 제목 (예: 6주차 실습2 — PyTorch 신경망 학습)",
  "blocks": [ {블록 spec}, ... ]
}
```

## 블록 spec 타입 (03_build_notion.py block_from_spec 과 동일)

| type | 필드 | 비고 |
|------|------|------|
| `h2` | `text`, `stage`? | 큰 섹션 제목 (파란 배경 자동). `stage`(정수)를 주면 제목 앞에 `[②]` 배지가 붙어 흐름 지도의 몇 단계인지 표시 (해당 단계 섹션에만; import·부록은 생략) |
| `flowmap` | `goal`, `stages[]` | **📍 흐름 지도**(맨 위). `goal`=한 문장 목적, `stages`=`[{n, name, data}]`. 주황 callout + 정렬된 code 지도로 렌더. `data`=그 단계 종료 시 '주인공 데이터'의 shape/상태 |
| `h3` | `text` | 소제목 |
| `p` | `text` | 본문. `**굵게**` `_이탤릭_` `` `코드` `` 지원 |
| `callout` | `text`, `emoji`, `color` | 강조 박스. color 기본 default_background, 팁은 `blue_background`, 핵심은 `orange_background`, 주의는 `red_background` |
| `num` | `items`[] | 번호 목록 |
| `bul` | `items`[] | 불릿 목록 |
| `code` | `code`, `language` | 코드 블록. language 기본 python |
| `eq` | `expression` | LaTeX 수식 |
| `quote` | `text` | 인용 |
| `divider` | - | 구분선 |
| `output` | `text` | **코드 실행 결과(출력)**. 회색 코드블록으로 렌더됨. 노트북 셀의 `outputs` 값을 그대로 넣어 "이 코드를 돌리면 이렇게 나와요"를 보여줄 때 |
| `image_ref` | `key` | **코드가 만든 그림**(matplotlib 등). 노트북 셀의 `output_images` 배열에 있는 key를 그대로 넣으면 구글드라이브 이미지로 자동 치환됨 |

## 출력·그림 사용 규칙 (중요)

노트북 cells의 각 code 셀에는 `outputs`(실행 결과 텍스트)와 `output_images`(그림 key 목록)가 들어있다.
- 코드를 설명한 직후, 그 셀에 `outputs`가 있으면 **`output` 블록으로 실제 결과를 보여주고** "이 결과는 ~를 의미해요" 해설을 붙인다.
- 그 셀에 `output_images`가 있으면 **각 key를 `image_ref` 블록으로** 넣어 그림을 보여주고, 그림이 무엇을 나타내는지 설명한다.
- key는 절대 지어내지 말 것. 노트북 데이터의 `output_images`에 있는 값만 사용.

## 흐름 지도(flowmap) 작성 규칙 — 숲 먼저

**왜**: 코드 0 기초자는 줄 단위 풀이(나무)만 보면 "지금 왜 이걸 하는지, 전체 어디쯤인지"를 잃는다.
맨 위에 흐름 지도로 **숲**을 먼저 보여준 뒤, 각 섹션에 단계 배지를 달아 항상 위치를 알려준다.
(세계적 프로그래머의 코드 읽기 2원칙 — ① 큰 그림 먼저 ② 데이터를 주인공으로 따라가기 — 를 산출물에 심는 장치.)

- **위치**: 🎯 목표 / 📋 선수지식 callout 다음, 첫 divider·섹션 직전 (맨 위 블록 중 하나).
- **`goal`**: 이 노트북이 결국 뭘 만드는지 **한 문장**. (예: "타이타닉 데이터로 생존을 예측하는 모델 만들기")
- **`stages`**: 노트북을 **5~8단계**로. 데이터/ML 노트북의 표준 뼈대에서 **실제 등장하는 것만** 고른다:
  데이터 불러오기 → 탐색(EDA) → 전처리(결측치·인코딩·스케일) → 특징/정답 분리 → train/test 분할 → 모델 정의 → 학습 → 평가 → 예측/시각화.
  노트북의 마크다운 셀 제목을 길잡이로 삼되, 없으면 코드 흐름으로 판단.
- **`data` (가장 중요)**: 각 단계가 끝났을 때 **'주인공 데이터'의 shape/상태**를 적는다(df/배열/텐서/model).
  예: `df (891×12)` → `X, y (891×8, 891×1)` → `model.fit(...)` → `정확도 0.82`.
  이 데이터 여정(shape journey)이 "코드를 데이터 흐름으로 읽기"를 가르치는 핵심 — 프로즈로 두지 말고 **반드시 shape로**.
- **`name`은 짧게** (한글 ~10자 이내) — monospace 지도가 깔끔히 정렬된다.
- **연결**: `flowmap`의 `stages[].n` 과, 그 단계 섹션 `h2`의 `stage` 값을 **똑같이** 맞춘다.
  h2 텍스트 앞에 `1.` 같은 수동 번호는 쓰지 않는다(배지와 중복).

```json
{"type": "flowmap",
 "goal": "타이타닉 데이터로 '생존 여부'를 예측하는 모델 만들기",
 "stages": [
   {"n": 1, "name": "데이터 불러오기", "data": "df (891×12)"},
   {"n": 2, "name": "결측치 처리",     "data": "df (891×12)"},
   {"n": 3, "name": "특징/정답 분리",   "data": "X, y (891×8, 891×1)"},
   {"n": 4, "name": "train/test 분할", "data": "6:4"},
   {"n": 5, "name": "모델 학습",       "data": "model.fit(X_train, y_train)"},
   {"n": 6, "name": "평가·시각화",      "data": "정확도 0.82"}]}
```
그리고 섹션마다: `{"type":"h2","text":"결측치 처리","stage":2}` → 제목이 `[②] 결측치 처리` 로 렌더된다.

## 가이드북 작성 원칙 (코드 0 기초자 대상, 전체 상세형)

1. **맨 위**: callout으로 "이 노트북으로 할 수 있게 되는 것" + 선수지식 → 이어서 **📍 흐름 지도(flowmap)** (목적 + 5~8단계 데이터 shape 여정; 위 "흐름 지도 작성 규칙" 절 참조)
2. **import 섹션**: 쓰인 라이브러리마다 "무엇을·왜" 한 줄 설명 (bul)
3. **코드 흐름을 의미 단위로 섹션화** (h2/h3). 노트북의 마크다운 셀을 길잡이로 활용. 각 섹션 h2에 흐름 지도의 단계 번호를 `stage`로 달아 `[②]` 배지로 잇는다.
4. **각 코드 조각마다**:
   - `code` 블록으로 실제 코드 제시 (노트북에서 발췌, 너무 길면 핵심만)
   - `p`로 **한 줄씩 무슨 뜻인지** 풀이 (코드 모르는 사람 눈높이)
   - 함수/메서드는 "`함수()` 는 ~를 한다" 식으로 문법 설명
5. **하이퍼파라미터/주요 인자**가 나오면:
   - `bul` 또는 표로 "인자 = 의미, 이 값이면 이렇게 됨, 권장 기준"
   - callout(blue_background)로 "⚙️ 실무 팁: 이 값을 어떻게 정하나"
6. **실무 적용**: callout으로 "💼 실무에서는 …" (언제 쓰나, 흔한 실수, 대안)
7. **맨 끝**: "✅ 핵심 체크리스트" (num) — 이 노트북에서 꼭 기억할 것
